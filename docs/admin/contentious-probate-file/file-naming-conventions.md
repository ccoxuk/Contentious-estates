# File Naming Conventions - Contentious Probate

<!-- 
  Purpose: Establish consistent document naming standards for easy identification and retrieval.

## Overview

Consistent file naming is essential for efficient document management. These conventions ensure documents are easily identifiable, properly versioned, and professionally organized.

---

## Core Naming Convention

### Standard Format

```
[Date YYYYMMDD]_[DocumentType]_[Subject/Party]_[Version].[Extension]
```

### Components Explained

**1. Date [YYYYMMDD]**
- Format: Year (4 digits), Month (2 digits), Day (2 digits)
- Example: 20250116 for 16 January 2025
- Purpose: Chronological sorting, quick date identification
- Always use document creation or primary date

**2. Document Type**
- Abbreviated descriptor of document type
- Use standard abbreviations (see list below)
- 2-6 characters typically
- Capital letters for clarity

**3. Subject/Party**
- Brief descriptor of content or recipient
- Camel case or underscores for multi-word
- Party name, document subject, or both
- Keep concise (1-4 words max)

**4. Version**
- DRAFT | v1 | v2 | v3 | FINAL | FILED | SERVED
- Indicates document status
- Essential for version control

**5. Extension**
- .docx - Word documents (editable)
- .pdf - PDFs (final/served versions)
- .xlsx - Excel spreadsheets
- .msg - Saved emails
- .jpg/.png - Images

## Version Control

### Version Descriptors

**DRAFT**
- Initial working draft
- Not for external use
- May be incomplete

**v1, v2, v3, etc.**
- Sequential numbered versions
- Track revisions and changes
- Use for internal review cycles

**FINAL**
- Approved version
- Ready for filing or service
- No further changes expected

**FILED**
- Filed with court
- Includes court file stamp or reference
- Archive previous versions

**SERVED**
- Served on parties
- Includes proof of service
- Final version for that purpose

**RECEIVED**
- Document received from opponent or third party
- Preserves original filename information

### Version Progression Example

```
20250120_POC_InheritanceActClaim_DRAFT.docx
20250121_POC_InheritanceActClaim_v1.docx (after supervisor review)
20250122_POC_InheritanceActClaim_v2.docx (after counsel's comments)
20250125_POC_InheritanceActClaim_FINAL.pdf (approved version)
20250127_POC_InheritanceActClaim_FILED.pdf (filed version with court stamp)
20250127_POC_InheritanceActClaim_SERVED.pdf (served version with service details)
```

## Email Naming

### Saved Emails

When saving emails to matter file:

```
20250116_EMAIL_FromClient_NewEvidence.msg
20250117_EMAIL_ToOpponent_Settlement.msg
20250118_EMAIL_FromCounsel_Advice.msg
```

### Email Attachments

Save attachments using standard convention, not original filename:
```
Original: "document final version 2.docx"
Save as: "20250116_WS_Witness_RECEIVED.pdf"
```

## Do's and Don'ts

### Do:

✓ Use consistent format for all documents  
✓ Include date in YYYYMMDD format  
✓ Use standard abbreviations  
✓ Include version descriptor  
✓ Be concise but descriptive  
✓ Use underscores for spaces  
✓ Review and correct before saving

### Don't:

✗ Use spaces in filenames  
✗ Use special characters (except underscore and hyphen)  
✗ Use vague descriptions like "letter" or "document"  
✗ Omit dates  
✗ Skip version control  
✗ Use inconsistent date formats  
✗ Make filenames too long (>100 characters)  
✗ Use client-sensitive info in filename if shared externally

## System Compatibility

### Character Restrictions

Avoid these characters in filenames:
- / \ : * ? " < > |
- Use underscore instead of spaces
- Avoid & % £ $ and other special symbols

### Length Limits

- Windows: Max 260 characters (full path)
- Keep filenames under 100 characters
- Keep folder depth under 10 levels

### Case Sensitivity

- Windows is not case-sensitive
- Some systems are case-sensitive
- Use consistent capitalization for clarity

## Quality Control

### Before Saving/Filing

Check:
- [ ] Date correct and in YYYYMMDD format
- [ ] Document type abbreviation correct
- [ ] Subject/party accurately described
- [ ] Version descriptor appropriate
- [ ] Extension correct (.docx for editable, .pdf for final)
- [ ] No spaces or special characters
- [ ] Filename length reasonable
- [ ] Consistent with firm conventions

## Training and Implementation

### New Staff

- Provide this guide
- Show examples from existing files
- Review first few document names
- Provide feedback and correction

### Existing Files

- Apply conventions to all new documents
- Rename key documents if time permits
- Don't rename everything (risk of broken links)

---

## Technology Integration

### Document Management Systems

- Use metadata in addition to filenames
- Take advantage of search functionality
- Enable version control features
- Use tagging for additional classification

### Templates

Pre-populate filename templates:
```
YYYYMMDD_LTR_[ClientName]_[Subject]_v1.docx
```

## Exceptions

### When to Deviate

Rarely, but acceptable when:
- Court requires specific naming format
- Client provides document with own naming
- Third party requires specific format
- Bundle pagination requires sequential numbering

**Document any exceptions in file note**
